Evolving Game

In my experience, integrating systems, people and processes has been a hard ball, and the game of it just keeps evolving. There are several dimensions around this term, but my experience has been in system integration in the enterprise ecosystem. However with cloud, mobility and social integration ways, we all have to just keep learning, but relaxation is necessary exercise. Please check out my earlier blog Let Just Integrate Blog Post for a brief.

Integration Timeline

  • File Based Batch Integrations Sevetys going into Eightys

    Gained popularity by pushing data feeds into giant monoliths like mainframes and even used today with FTP on B2B integrations. However architects were relying on Batch Processes to approach integration design.

  • Two Tier Architectural Style, MOMs, Real-Time Synchronous Eightys going into Nintys

    Along came in the 1990s, Application Architecture with 2-tiered architectural style developed with data residing in Oracle DB and Microsoft SQL Server, IBM Message Oriented Middleware, MQ Series and monoliths like ERPs like SAP Oracle, Warehouse Management System & Financial & Accounting Systems. Architects start to approach integration design with Real-Time Processes and interface based RPC style with broker based CORBA architectures.

  • ERPs, Web 1.0, Multi-Tier Architecture with Sun JAVA, Microsoft .NET Stacks Nintys going to 2000

    Over the past few decades, an interesting Enterprise and Application Integration landscape was developed with databases like Oracle, SQL Server, Sybase persisting trillions of data records, Packaged Applications like SAP, Oracle, JD Edwards, PeopleSoft becoming system of records in enterprises with rich User Interfaces, Internet and Web based architectural style gained popularity with Java & .Net stack.

  • EAI, Point to Point, Broker Based Hub & Spoke Five years in 20th Century going to 2005

    Architects, both business and technical, coined a term Enterprise Application Integration (EAI) and Business Process Integration (BPM) to address integration needs and challenges in ever interesting integration landscape.

    Niche integration vendors like WebMethods and TIBCO created products that integration consultants learned and started delivering on integration scenarios for clients using these tools and consulting mindset. However big enterprises and internet companies started enabling SOA architectures with interoperable services using the SOAP protocol and data exchange with XML, and clients started deploying on Application Servers like Weblogic, Websphere based on Java EE6.

  • Enterprise Service Bus 2005 going to 2010

    Bus pattern started to emerge where architects analyzed that a lot of application and business logic is being developed in the middle tier and though it provides speedy services, we end up with criss cross connections and overtime, middleware becomes monoliths which are difficult to maintain and we have to correlate across application & transaction boundaries. In spite of centralized middleware, we will be better off with a service bus that is based on components, uses Domain Specific Language and not just transformation language like XSL / XSLT. Application/Services connects with the Service Bus on edges and integration developers leverages service bus capabilities to address integration design tasks like validation, enrichment, transformation, routing. ESB products are usually runtimes with low memory footprints which are either provisioned along with Application Server like Oracle Service Bus with Weblogic, or developed with open source community way like Mule ESB, Apache Camel with enterprise licensing backed by organizations like Mulesoft, Red Hat.

  • Cloud Computing,Mobility, Social, IOT 2010 going to 2020

    We know about hypervisor, virtual servers but it is very different when you can provision a server with a click and choose the hardware configuration with a template. Cloud computing basically automated hardware provisioning with software tools and implementation of elastic design moved architecture layers into the cloud (IAAS, PAAS and SAAS). Products developed with cloud native architecture saw huge growth and all of these products are backed by APIs and platforms.

    Web interestingly is not the only client after the invention of the iPhone, and it is very different when there are platforms connecting millions of people. Smartphones are also enabling smart devices and there are billions of devices connected on the network which is called Internet of Things.

  • Microservices, API 2010 going to 2020

    Implementation stack at platform companies like Netflix explained that instead of monoliths, we have to develop components and services modeled around business domains that can be scaled and deployed independently.
